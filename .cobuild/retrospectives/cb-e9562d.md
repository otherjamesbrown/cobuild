## What Worked

1. **Design-first approach.** Writing the gap analysis as a design shard with structured phases/tasks made implementation straightforward. Every task had clear scope.

2. **Wave decomposition.** Dependencies were correctly identified. No task was blocked by a missing predecessor.

3. **Eating our own cooking.** Running CoBuild commands to manage CoBuild's own development surfaced real bugs immediately. Every bug found was a bug that would have hit the first real user.

4. **Connector abstraction.** Once the CP connector's --project bug was fixed, `cobuild wi` commands worked seamlessly for both cobuild (cb-) and penfold (pf-) projects.

5. **Interactive design review.** Having the developer review the design before submission caught MEDIUM issues early. The formal gate would have passed anyway, but the human review added value.

## What Failed

### Critical bugs blocked the entire pipeline

9 bugs were discovered, 5 of which were blocking:

| # | Bug | Phase blocked | Root cause |
|---|-----|---------------|-----------|
| 1 | CP connector missing --project | All write operations | run() didn't pass global flags |
| 2 | Wrong column name (agent → creator) | Readiness gate | Legacy code assumed wrong schema |
| 3 | Missing shard ID generation | Readiness gate | Table has no default, client didn't generate |
| 4 | Nil client panic | Pipeline init | Agent identity removed but client still required it |
| 5 | Missing .cobuild.yaml | Connector init | Repo didn't have project config |

**Pattern:** 4 of 5 critical bugs were in the **legacy client** layer (direct Postgres access) — the exact layer that Phase C of the design targets for migration. The connector abstraction was correct; the legacy code underneath was broken.

### Decomposition didn't register tasks

Tasks were created as work items via `cxp shard create` but the decomposition step didn't call `cbClient.AddTask()` to register them in `pipeline_tasks`. This means:
- `cobuild status` showed 0 tasks
- No dispatch/completion timestamps in the DB
- Data had to be backfilled manually

**Fix needed:** The decompose skill or the `cobuild gate decomposition-review` command should register all child tasks in `pipeline_tasks` when the gate passes.

### Manual mode bypassed tracking

Because we used `cobuild wi status` to update tasks instead of going through `cobuild dispatch` → `cobuild complete`, we lost:
- Dispatch timestamps
- Agent session IDs
- Worktree paths
- Build/test evidence
- PR URLs

This is expected for a manual run, but it means the retrospective has less data to work with.

### Review shard IDs had wrong prefix

Gate review shards were created as `-811bd0` instead of `cb-811bd0` because the prefix logic was wrong. This is cosmetic but makes the audit trail harder to follow.

## Suggested Changes

### Skills
- **[decompose/decompose-design.md]** Add step: "Register tasks in pipeline via cobuild" after creating work items. Currently the skill only creates work items, not pipeline tracking records.
- **[done/gate-retrospective.md]** Add output format: write retrospective to `.cobuild/retrospectives/<design-id>.md` in addition to storing as a work item. Include data from pipeline_tasks and pipeline_gates tables.
- **[shared/design-review.md]** The MEDIUM finding about "autonomous error handling" was deferred. Should be a separate design, not left hanging.

### Config / Commands
- **`cobuild decompose --verdict pass`** should auto-register child tasks in pipeline_tasks when the decomposition gate passes.
- **`cobuild status`** should show task counts from the work-item system (via connector) when pipeline_tasks is empty, as a fallback.
- **`cobuild retro`** should auto-generate the retrospective markdown and offer to store it.

### Process
- **Run the formal gate, not just the interactive review.** The design review caught the same things the gate would, but didn't record them in the audit trail.
- **Use `cobuild dispatch` even in manual mode** to get dispatch timestamps and evidence.
- **Phase C (legacy client migration) is urgent.** 4 of 5 critical bugs were in the legacy layer. Every new feature built on top of it risks similar issues.

## Conclusion

The pipeline's manual mode is functional end-to-end. The primary risk area is the legacy client layer — it accounts for the majority of critical bugs and is the most fragile part of the system. Phase C (connector+store migration) should be prioritized.

The design-review → gate → decompose → implement → merge flow works, but the tracking data is only as good as the commands used. Bypassing `dispatch`/`complete` loses data. The decompose step needs to register tasks in both the work-item system AND the pipeline tracking table.

For a first run, this went well — 7 tasks completed, 9 bugs found and fixed, pipeline commands verified working across two projects.
