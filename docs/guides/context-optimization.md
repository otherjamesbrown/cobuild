# Context Optimization

A practical guide to keeping dispatch context lean. If you're here because `cobuild dispatch` warned that your context is too large, start at [Triage a warning](#triage-a-warning).

Companion guide: [Context Layers](context-layers.md) (what layers are and how they're configured).

## Why context size matters

Every dispatched agent receives an assembled `dispatch-context.md` built from your `.cobuild/context/` layers plus any config-driven layers. Large or polluted context causes several failure modes we see repeatedly:

- **Agent confusion** — the real task prompt gets buried under verbose file listings and stale notes, leading to off-topic edits and loops.
- **Wasted tokens** — every dispatch pays for every byte, whether the agent reads it or not.
- **Cache misses** — agents that repeatedly re-read the same pages rotate earlier content out of their working set.
- **Slower first turn** — large prefixes make the first tool call slower and the failure loop tighter.

Empirically, dispatched sessions over **30 KB** of assembled context start showing these symptoms; over **100 KB** the degradation is severe.

## Thresholds

CoBuild uses these as its warning levels (see `internal/contextaudit/audit.go`):

| Signal | Size | Meaning |
|--------|------|---------|
| Per-file `oversize` | ≥ 15 KB | Worth a second look — usually trimmable. |
| Per-file `very-large` | ≥ 30 KB | Almost always redundant or bloated. |
| Assembled `warn` | ≥ 30 KB | Pre-dispatch warning fires; `cobuild context audit` recommended. |
| Assembled `high` | ≥ 100 KB | Strong correlation with degraded output. Trim before dispatching real work. |

## Triage a warning

When `cobuild dispatch` emits `Warning: dispatch context is N KB`:

1. Run `cobuild context audit` to see the per-file breakdown.
2. Focus on the largest files first — the top 2–3 usually account for most of the bloat.
3. Look for specific bloat patterns (below).
4. If the flagged file is `anatomy.md`, investigate what `cobuild scan` included that it should not have.

## Common bloat sources

### 1. `anatomy.md` picking up cache dirs or binary data

`cobuild scan` generates `.cobuild/context/always/anatomy.md` as a file index. It should skip cache directories and large data files, but sometimes project layout defeats its heuristics. If `anatomy.md` contains references like `.pytest_cache`, `__pycache__`, `.ruff_cache`, `node_modules/`, `data/raw/`, or `.venv/`, those paths are noise.

**Fix:** add the directories to scan's skip list and re-run `cobuild scan`. The `cache-pollution` flag from `cobuild context audit` points at this case.

### 2. Phase-specific content in `always/`

`always/*.md` loads for every phase — design, implement, review, gates. Content that only matters during implement (e.g. build commands) wastes tokens during design and review.

**Fix:** move the content into `.cobuild/context/implement/` (or whichever phase needs it). Agents in other phases stop paying for it.

### 3. Duplicated CLAUDE.md content

If your pipeline config also loads `claude-md`, your project's `CLAUDE.md` is already part of the context. A layer file that restates the same rules doubles the bill for no benefit.

**Fix:** delete the duplicate layer file, or trim it to the parts CLAUDE.md does not already cover.

### 4. Verbose runbooks that belong in skills

Long runbooks for specific actions (how to cut a release, how to roll back) belong in skills, not context layers. Skills load on demand; context layers load on every dispatch.

**Fix:** convert the runbook into a skill (`skills/<name>.md`) and reference it from the dispatch prompt.

### 5. Historical notes

"Context for how we got here" often grows as a session log in `.cobuild/context/`. Agents need *current* state, not chronology.

**Fix:** delete, or archive to a commit message or a shard.

## Per-phase layout

A well-organized `.cobuild/context/` tree looks like this:

```
.cobuild/context/
├── always/
│   ├── anatomy.md       # auto-generated by `cobuild scan`
│   └── architecture.md  # high-level; kept under 10 KB
├── design/
│   └── ...              # only what design-phase agents need
├── implement/
│   └── ...              # build, test, commit conventions
└── review/
    └── ...              # review rubric, gate criteria
```

Content goes in the narrowest bucket that makes it available where it's needed.

## Checking your work

After trimming, re-run `cobuild context audit`. The goal is:

- No file flagged `very-large` (≥ 30 KB).
- On-disk total under 30 KB if possible, under 100 KB always.
- `anatomy.md` small enough to be useful (typically 5–15 KB for a medium repo).

Run a dry-run dispatch to confirm no warning fires:

```
cobuild dispatch <task-id> --dry-run
```

## How context size is tracked

Every dispatch records the assembled context size in `pipeline_sessions.dispatch_context_chars`. You can query it directly:

```sql
SELECT design_id, phase, runtime, dispatch_context_chars
FROM pipeline_sessions
WHERE started_at > NOW() - INTERVAL '7 days'
ORDER BY dispatch_context_chars DESC
LIMIT 20;
```

Designs that consistently dispatch with large contexts are candidates for trimming — or for breaking into smaller tasks with narrower context.
