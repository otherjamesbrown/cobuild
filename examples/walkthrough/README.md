# CoBuild Walkthrough Example

This directory is a miniature CoBuild repo root stored inside the main
`cobuild` repository. Start by reading it in place:

```bash
cd examples/walkthrough
```

From there, treat the current directory as if it were the root of a small
project. The files here show the minimum moving parts for a design-to-review
pipeline, plus the deploy handoff that follows review. For the full product
docs, use this as the short tour and keep the canonical references close:
[docs/cobuild.md](../../docs/cobuild.md),
[docs/guides/config.md](../../docs/guides/config.md),
[docs/guides/skills.md](../../docs/guides/skills.md), and
[docs/guides/audit-trail.md](../../docs/guides/audit-trail.md).

## What is in here

- `.cobuild.yaml` gives the project identity and shard prefix.
- `.cobuild/pipeline.yaml` shows a deliberately small pipeline config with one
  connector, one runtime in the file, the standard workflow shapes, and one
  example deploy target.
- `.cobuild/context/always/anatomy.md` is auto-loaded on every dispatch.
- `skills/design/gate-readiness-review.md` is a repo-local skill file.
- `work-items/` holds the fictional shard bodies that the walkthrough refers to.
- `artifacts/gate-verdict.readiness-review.json` is a parked copy of the
  temporary verdict file a gate agent would normally write to
  `.cobuild/gate-verdict.json`.

## The fictional shard set

The parent design is [wx-1000-design.md](work-items/wx-1000-design.md). It
describes a tiny feature for a fictional `demo-service`: expose a release
banner through an API, render it in the UI, and verify it during deploy.

The decompose step would split that design into three child task shards:
[wx-1001-task-api.md](work-items/wx-1001-task-api.md),
[wx-1002-task-ui.md](work-items/wx-1002-task-ui.md), and
[wx-1003-task-smoke.md](work-items/wx-1003-task-smoke.md).

## What `cobuild orchestrate wx-1000` would do

If you loaded these shard bodies into a real connector and then ran
`cobuild orchestrate wx-1000`, CoBuild would auto-initialize the pipeline at
the design phase and start walking it forward.

In `design`, the dispatched gate agent would read the design shard body, load
the always-on context from
[anatomy.md](.cobuild/context/always/anatomy.md), and also load the repo-local
skill from
[gate-readiness-review.md](skills/design/gate-readiness-review.md) because
`.cobuild/pipeline.yaml` adds it as a `phase:design` context layer. The agent
would decide whether the design is specific enough to split into tasks, then
write `.cobuild/gate-verdict.json`.

In `decompose`, CoBuild would turn the approved design into task shards like
the three sample task files in `work-items/`. That is the point where shard
boundaries and dependencies get created. In this example, the UI task depends
on the API task, and the smoke/deploy task depends on both.

In `implement`, CoBuild would dispatch each task in its own worktree. The
prompt for each implementing agent would be assembled from three places: the
auto-discovered always context, the `dispatch-prompt` layer, and the
`parent-design` layer. The sample config uses harmless `printf` commands for
build and test so the example stays file-only.

In `review`, the implementation PRs would be reviewed according to the review
strategy in `.cobuild/pipeline.yaml`. When review passes and the PRs merge, the
design advances to `done`.

Deploy sits just after that loop. The example config includes one deploy target
so you can see where deploy rules live. In a real run, `cobuild orchestrate`
would stop at the human approval point, you would inspect
`cobuild deploy wx-1000 --dry-run`, and only then approve `cobuild deploy
wx-1000`.

## Where the prompt comes from

The shortest way to think about prompt assembly is:

1. Auto-load any markdown files under `.cobuild/context/always/`.
2. Add any extra layers from `.cobuild/pipeline.yaml`.
3. Inject shard-specific content such as the dispatch prompt and parent design.

That means this walkthrough's design-phase prompt is the combination of the
always-on anatomy note, the repo-local readiness skill, the dispatch prompt
generated by CoBuild, and the connector-returned shard body. The broader rules
for context assembly are in
[docs/guides/context-layers.md](../../docs/guides/context-layers.md).

## How `gate-verdict.json` becomes a gate record

During a gate phase, the dispatched agent writes a JSON file at
`.cobuild/gate-verdict.json`. The parked example in
[artifacts/gate-verdict.readiness-review.json](artifacts/gate-verdict.readiness-review.json)
shows the exact shape, including the structured `readiness` field.

When the gate session exits, the runtime reads that file and hands the verdict
back to the appropriate gate flow. For a readiness review, that becomes a gate
record on the design. For PR review, the same pattern feeds the review gate for
the task. The audit trail is then visible through:

```bash
cobuild audit wx-1000
cobuild show wx-1000
```

Use `cobuild audit` for the timeline and `cobuild show` for the current phase,
task list, and links to any review sub-shards. The deeper audit model is
documented in [docs/guides/audit-trail.md](../../docs/guides/audit-trail.md).

## Why this example is in-tree

This walkthrough lives inside `cobuild` so it can be versioned and tested with
the product. If you want to try it as a real standalone repo, copy the
directory into its own git repository and keep the same file layout at the repo
root.
